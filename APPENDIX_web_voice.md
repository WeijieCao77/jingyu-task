# 附件 · 网页版语音入园（备选方案，最终未选用）

> 这是我们**做出来并验证可用**、但**最终方案没有采用**的一条路径，归档在此作为能力佐证。
> 最终选用的是**电话方案**（任意手机拨号即接通 AI 门卫）。

## 它是什么
浏览器内的 WebRTC 实时语音页面，访客**不用打电话**，直接在手机/电脑网页里点一下就和 AI 门卫对话：
- `GET /voice` —— 访客语音页（点"接入"→ 授权麦克风 → 和 AI 实时对话登记）。
- `GET /qr` —— 一张可打印的二维码海报，扫码直达 `/voice`（贴门口"扫码即用"）。
- `GET /token` —— 为浏览器签发 LiveKit 加入令牌；`GET /guard_call` —— 门卫加入访客房间转人工。
- 代码：`src/visitor_agent/web/server.py` 里的 `_VOICE_HTML` 及上述路由（**保留未删**，随时可演示）。

## 状态
✅ 已实现、可用。上云后走 LiveKit Cloud + https，任意网络的浏览器都能用（本地 http 时受麦克风权限限制，上云后解决）。
现在仍可直接访问：`https://web-production-d105c.up.railway.app/voice`（允许麦克风）或 `/qr`。

## 为什么最终没选它
- **电话更贴近真实门岗**：访客到门口直接拨号即可，无需扫码、无需打开网页、无需授权麦克风。
- **门禁场景**：电话是所有人都会用的最低门槛入口；网页语音要求 HTTPS + 浏览器麦克风授权，门口体验不如电话直接。
- 因此最终主线 = **电话**（Twilio → LiveKit Cloud → 云端 AI 门卫）；网页语音作为备选能力保留。

## 如需启用/演示
无需任何改动，直接打开 `/voice` 或把 `/qr` 打印出来即可——两套入口（电话 + 网页语音）共用同一个 agent 与同一套登记/通知/查询逻辑。
