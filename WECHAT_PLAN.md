# 微信推送预备方案（3 天内可落地）

> 需求：访客与 AI 完成问答后，把访客信息**直接发到保安的微信**；保安在微信里查看、核对后，**回复或点按钮放行**。
> 现在不急着实现，本文是"要做时照着做"的预备方案 + 取舍 + 时间表。系统里通知渠道已是**可插拔**的（`notify/dispatch.py`），接微信只是新增/启用一个渠道，不动主流程。

> **定位（已定）**：当前/demo 初步用 **Telegram**（见 `NOTIFY_SETUP.md`，最快到手机）；**企业微信/微信是国内生产的最终落地目标**——demo 变产品、在国内上线时，通知必须落到国内软件。本文即"切到企微"的落地方案。

## 零、你需要准备什么？（不需要营业执照 / 不需要真公司）

**结论：个人用手机号就能免费创建企业微信，跑通方案 A，无需营业执照、无需企业认证。**

| 问题 | 答案 |
|---|---|
| 需要一家真公司吗？ | **不需要**。创建组织时填个名字即可（如"蓝色鲸鱼demo"），可跳过认证。 |
| 需要营业执照吗？ | **方案 A（群机器人）不需要**。营业执照只在"企业认证"时才要——认证是可选的，用于扩大成员上限/高级功能。 |
| 你之前没用过企微？ | 没关系，下面 4 步即可。 |

**方案 A 的准备步骤（约 10 分钟）：**
1. 手机应用商店装 **企业微信** App（或电脑访问 work.weixin.qq.com）。
2. 注册 → **创建企业/组织** → 用手机号 + 填个组织名 → **跳过/暂不认证**。
3. 在企业微信里建一个**群**（把你自己加进去，自己一个人也行；真用时把保安加进去）。
4. 群 → 右上 … → **群机器人 → 添加 → 复制 Webhook 地址**。
   把它填进 `.env`：`NOTIFY_CHANNEL=wecom`、`WECOM_WEBHOOK_URL=<地址>`。

> 注意：消息是发到**企业微信 App 里的那个群**，所以"看消息的人"（你/保安）手机要装**企业微信**（不是普通微信）。
> 这也是为什么 demo 先用 Telegram——它更轻、不用装企微。企微留给国内正式落地。

**方案 B（自建应用·原生按钮）需要更多：** 仍可个人创建，但配"可信域名/回调 URL、消息加密"较繁；部分能力可能要企业认证（需营业执照+对公账户）。所以方案 B 留给产品化阶段，demo 不碰。

## 一、两种实现，按"3 天内能稳的"排序

### 方案 A（推荐先上）：企业微信群机器人 Webhook + 链接放行 —— 0.5~1 天
- **推送**：建一个企微群（保安们在里面），加"群机器人"拿到一个 Webhook URL，后端 POST 一张 markdown 卡片（车牌/单位/事由/手机/姓名/入场时间）。**这套代码已经有了**（`notify/wecom.py`），把 `NOTIFY_CHANNEL=wecom` + `WECOM_WEBHOOK_URL` 填上即用。
- **放行**：卡片里带一个**带 token 的链接**「✅ 确认放行」，保安在企微内置浏览器点一下 → 打到后端 `/confirm?token=…` → 校验后抬杆。**这套也已经有了**（`web/server.py` 的 `/confirm`）。
- **优点**：秒级建群、个人手机号即可注册免费版、无需营业执照、国际可用、当天能通。
- **局限**：群机器人是**单向推送**，没有"消息内按钮回调"，所以放行用的是链接而非原生按钮（点链接打开网页确认）。对 demo/小园区完全够用。

### 方案 B（要"微信里原生按钮 + 回复"时）：企业微信自建应用 —— 2~3 天
- 在企微管理后台建一个**自建应用**，用**应用消息 + 模板卡片（template_card）**，卡片上带原生「放行 / 转人工」按钮；保安点按钮 → 企微回调你配置的服务器 URL → 后端校验后抬杆。还能接收保安的**文字回复**。
- **优点**：体验最接近"微信里直接审核、点按钮、回复"。
- **成本**：要配自建应用、可信域名/回调 URL（HTTPS 公网可达）、处理加密回调（AES）、access_token 刷新。约 2~3 天。

> 结论：**3 天目标 = 方案 A 当天先通（已具备），剩余时间按需升级到方案 B 的按钮回调。**

## 二、落地清单（方案 A，要做时照这个走）
1. 企微建群 → 群机器人 → 复制 Webhook URL。
2. `.env`：`NOTIFY_CHANNEL=wecom`、`WECOM_WEBHOOK_URL=<URL>`、`PUBLIC_BASE_URL=<公网HTTPS>`（保安点链接要可达——云部署见 DEPLOY.md，或隧道）。
3. 跑一遍：完成一次登记 → 群里收到卡片 → 点「确认放行」→ 后端抬杆、Dashboard 状态变已放行。
4. （可选）卡片内容/样式在 `notify/wecom.py` 的 `build_markdown` 调。

## 三、落地清单（方案 B，升级到按钮回调，要做时新增一个渠道）
1. 企微后台建自建应用，拿 `corpid` / `agentid` / `secret`；配可信回调域名 + 回调 URL + Token/EncodingAESKey。
2. 新增 `notify/wecom_app.py`：用 `access_token` 发**模板卡片**（按钮 = 放行/转人工，`task_id` 关联本次访客）。
3. 新增 `web/server.py` 的回调端点 `/wecom/callback`：验签 + 解密企微回调 → 解析按钮事件 → 调 `repo.mark_confirmed_by_id` + 抬杆；保安文字回复也走这里。
4. `NOTIFY_CHANNEL=wecom_app`，dispatch 里加一支路由。

## 四、与现有架构的契合点（为什么 3 天能成）
- **通知是适配器**：`notify/dispatch.py` 按 `NOTIFY_CHANNEL` 路由，加微信渠道不碰对话/数据/Dashboard。
- **放行已抽象**：`/confirm?token=`（链接）和 `/api/confirm/{id}`（后台按钮）都已存在，方案 B 的回调复用同一套 `mark_confirmed + open_gate`。
- **数据已落库**：每条访客 + 开闸都在 `visits`，微信只是多一个"看到 + 放行"的入口。

## 五、个人微信路径：QClaw / OpenClaw（任务允许"个人微信亦可 Demo"）
- 任务明确"推送保安微信（个人微信或企业微信均可作为 Demo）"，并点名让评估 **QClaw / OpenClaw**。
- QClaw = 腾讯基于开源 **OpenClaw** 的产品：**扫码绑定个人微信**，可在微信里收发消息/语音、触发本地 agent。可把放行卡片推到**个人微信**。
- 取舍：itchat/wechaty 等第三方个人微信自动化违反腾讯 ToS、封号风险高，**不用**；QClaw 是腾讯官方、相对可控，可作 demo 的"真·微信触达"。但**生产仍建议企业微信**（合规 + 原生按钮回调=方案 B）。

## 六、合规与风险（务必知道）
- **第三方个人微信自动化（itchat/wechaty）不用**：违反腾讯 ToS、封号风险高。要个人微信触达走官方 QClaw。
- 方案 B 的回调 URL 必须 HTTPS 公网可达（云部署天然满足）。
- 手机号等个人信息，注意脱敏/留存期（见 PRODUCT_FLOW 决策点 #6）。
