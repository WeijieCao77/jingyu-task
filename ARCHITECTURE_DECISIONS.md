# 架构决策记录（ADR）：为什么这么做 · 优势 · 优势的根因 · 是否跟模型相关

> 这份文档把**整个迭代过程里我做的每一个架构决策**梳理成一条主线，并对每条回答四个问题：
> **① 为什么这么做 ② 优势是什么 ③ 这个优势的根因（为什么会有这个优势）④ 跟模型是否相关。**
> 逐版本流水见 `CHANGELOG.md`，需求账本见 `ITERATION_SUMMARY.md`，框架对照见 `FRAMEWORK_RESEARCH.md`。
> 本文是给**技术答辩**用的"判断过程"总结。

---

## 0. 一条主线（贯穿所有决策的设计哲学）

**"大脑（模型）是一个可替换的配置项；其余一切都是适配器。"**

我刻意把**模型**和它周围的所有东西解耦：编排框架、接入形态（电话/扫码/浏览器）、数据存储、通知渠道、抬杆执行、鉴权、查询工具——全部做成可替换的适配器，模型只通过明确的接口（function tools / OpenAI 兼容协议）参与。

这条主线直接决定了本项目优势的"模型相关性"分布：

- **大多数优势是架构性的、与具体模型无关**：并发、三种接入统一、通知多渠道、数据准确性的确定性部分、鉴权、持久化、转人工、放行闭环——换任何 LLM 这些都成立。
- **少数优势依赖模型能力**：realtime 低延迟、自然对话、AI 复述确认、对话式数据查询——这些需要 LLM 的对话/工具调用/语音能力。
- **而"模型可换"这一层，正是为了让上面那"少数依赖"不被单一模型锁死**：模型更强则这些能力更好，模型不可用则有确定性兜底或回退。换句话说，**我用工程手段给"模型能力"设界与兜底，让关键属性（准确性、安全性、可靠性、并发）不依赖模型，把模型用在它擅长且可替换的地方。**

下面逐条展开。每条标注【与模型关系】：**无关 / 相关 / 相关但有确定性兜底 / 元层面（关于模型本身）**。

---

## A. 编排框架：选 LiveKit Agents（不自己拼 SIP）

- **为什么**：任务核心是"打电话进来对话"。电话需要 SIP 接入、实时媒体、打断（barge-in）、多路并发、还要能转人工。自己用 WebSocket/媒体流拼这些既慢又脆。
- **优势**：电话/打断/并发/转人工"开箱即用"；一套 Agent 同时吃电话、扫码、浏览器三种接入（都汇入同一个 Room）。
- **根因**：LiveKit 原生提供 SIP↔WebRTC 桥、每通来电一个独立 job（无共享可变状态）、Room 抽象统一了所有接入形态。我没有把"语音 Agent 能力"外包给 SaaS（Vapi/Retell），而是留在自己手里——因为**这正是本题要考察的能力**。
- **【与模型关系】无关**。框架是传输与编排层，不是大脑；换任何 LLM，这些能力都不变。对比见 `FRAMEWORK_RESEARCH.md`（为什么不用 Pipecat/TEN/Vapi/Retell）。

## B. 语音架构：从三段 Pipeline → 默认 realtime（speech-to-speech）

- **为什么**：真机实测 pipeline 模式"卡一会"——首句慢、每轮有停顿。日志定位到两点：每通来电冷启动新进程（spawn+加载 VAD），以及本地 VAD 在模拟层下慢。
- **优势**：realtime 把首句延迟降到 **≈1.4s**，明显更快，是稳过"25 秒"的关键。
- **根因**：speech-to-speech 把 STT→LLM→TTS 三段串联合并为一个模型（音频进、音频出），**省掉了三段各自的排队延迟**；turn detection 在服务端做，**绕开本地 VAD 冷启动**。pipeline 的延迟则是三个模型延迟之和。
- **【与模型关系】强相关，且这是全项目"模型相关性"最典型的一处**。这个低延迟优势**直接来自 `gpt-realtime` 这类 s2s 模型**——没有 s2s 模型就没有这个优势。所以我**没有写死**：`VOICE_MODE` 一行在 `realtime`（默认，需该能力的模型）和 `pipeline`（回退，便宜、有文字、不依赖 s2s）之间切换。**模型决定了能不能拿到这个优势，架构决定了拿不到时能否优雅回退。**（详见 `ARCHITECTURE_AB.md`）

## C. 模型解耦：LLM_BASE_URL 接任意 OpenAI 兼容端点 + STT/TTS env 可切

- **为什么**：最终用哪个模型是客户定（含国内合规要求）。不能把项目焊死在某一个模型/厂商上。
- **优势**：**零代码**切换大脑——GPT / Claude / Gemini / Qwen / DeepSeek / 豆包 / Kimi 等任意 OpenAI 兼容端点；STT 可切 Deepgram、TTS 可切 Azure 中文音色。
- **根因**：统一走 OpenAI 兼容协议 + 把 provider 选择全部下沉到 `providers.py` 一个装配点 + 配置化（`.env`）。Provider 是适配器，不是硬编码。
- **【与模型关系】元层面（这条决策"关于模型本身"）**。它本身不提供业务优势，但它是让 B/D/H 里"模型相关的优势"**不被单一模型锁定**的基础设施：模型更强→自动更好；某模型不可用/不合规→换一个，主流程不动。**这是把"模型依赖"从风险变成可控变量的关键一招。**

## D. 数据准确性：分场景策略（这是"削弱模型依赖"的核心思想）

语音识别号码、单位最容易错。我没有寄希望于"换个更强的模型识别得更准"，而是**按数据类型分而治之**：

- **D1 开放集合（车牌号、手机号）→ AI 复述确认层**
  - 为什么：号码是开放集合，没有名单可查；STT 必然有错。
  - 优势：登记前 AI 逐位念车牌、分组念手机让访客确认，错了就改——**把准确性的最后一道关交给人**。
  - 根因：用对话把"人确认"嵌进流程。
  - 【与模型关系】**相关但有确定性兜底**：复述、理解"对/不对"依赖 LLM 对话能力；但**最终正确性由"人确认"保证，而非模型识别率**——所以即使换一个识别略差的模型，结果仍正确。这正是"用产品设计削弱模型依赖"。

- **D2 封闭集合（来访单位）→ 公司名单模糊匹配（`roster.py`）**
  - 为什么：园区公司是封闭、已知的集合（如"蓝色金鱼"应是"蓝色鲸鱼"）。
  - 优势：把听错的单位**确定性地**纠正到官方名（金鱼→鲸鱼，拼音几乎相同）。
  - 根因：汉字相似度（difflib）+ 拼音相似度（pypinyin）打分，超阈值即纠正。**纯算法，不调模型。**
  - 【与模型关系】**完全无关**。封闭集合的准确性被我从模型手里拿走，交给确定性匹配。

- **D3 车牌省份归一（`slots.normalize_plate`）**：省份名→简称（广东A→粤A，31 省映射表）。**确定性，与模型无关。**

- **D4 黑白名单（`access.py`）→ 精确匹配**：车牌/手机规范化后**精确**匹配，黑名单优先（fail-safe）。**确定性，与模型无关**——访问控制这种安全决策绝不能交给概率模型。

> **D 的统一思想**：开放集合用模型+人确认，封闭集合/安全决策用确定性算法。**关键属性（准确、安全）尽量不依赖模型。**

## E. 回访识别画像（`repo.recognize`：手机=人，车牌=车）

- **为什么**：加分项要求"对回访者直接确认而非从头采集"；且要考虑真实场景（换车、换司机）。
- **优势**：按"手机=人、车牌=车"的身份模型分级识别——同车同人=自信确认，仅手机=认出本人（可能换车），仅车牌=认出车但不假设同一人；给出累计次数、上次单位/事由。
- **根因**：确定性的 DB 查询 + 身份建模；把"上次信息"作为"一句话确认"抛回，不盲目放行。
- **【与模型关系】无关**（识别是 DB 查询）/ 用一句话确认那步是 LLM（相关但非关键）。

## F. 接入形态：电话 / 扫码 / 浏览器三合一 + 主叫号预填 + 并发

- **F1 三种接入汇入同一 Room/Agent**
  - 为什么：任务要电话；但 demo 期浏览器/扫码更易测。不想为每种接入写一套逻辑。
  - 优势：**写一次 Agent，三种接入都能用**；电话路打通后，复述确认/名单/黑白名单/回访/转人工/放行播报全部自动复用。
  - 根因：LiveKit Room 抽象 + 业务逻辑全在 `session_logic`（与接入形态无关）。
  - 【与模型关系】**无关**。

- **F2 电话拨入：Twilio → LiveKit SIP → 自动派发 agent**（`scripts/setup_sip.sh`）。每通来电独立房间=天然并发。**与模型无关**（传输层）。

- **F3 主叫号预填手机**：SIP 来电的 caller-ID 就是访客手机，自动填进手机槽——省一问、免听错、立刻触发回访/黑白名单匹配。**与模型无关**（来自 SIP 元数据，不是模型推断）。

- **F4 多路并发**：多车同时拨打=多个独立 job，无共享可变状态。**与模型无关**（框架能力）。

## G. 放行闭环：通知多渠道 + 转人工 + FR-2 语音播报 + 抬杆

- **G1 通知可插拔（`notify/dispatch.py`：Telegram/企微/Discord，逗号可多推）**
  - 为什么：demo 要美国可用且秒到手机（Telegram），生产要国内落地（企业微信）；一度想绑死企微，后改为可插拔。
  - 优势：换/加渠道不动对话/数据/Dashboard 主流程；还能同时多推。
  - 根因：适配器模式，渠道只是一个"发卡片"的实现。
  - 【与模型关系】**无关**（纯工程）。

- **G2 转人工两模式**（管理员主动 + 客户要求）：保安加入同一 Room，AI 说一句让位后退出。**与模型无关**（框架同房间能力）。

- **G3 FR-2 放行后 AI 语音通知访客**：保安放行→web 向访客 Room 发 `{"type":"approved"}` 数据消息→agent 监听后语音播报"已放行请进"。**机制与模型无关**（跨进程数据消息）；播报这句话用 `_speak`（realtime/pipeline 自适应）。

- **G4 抬杆双模式**（`notify/gate.py`）：默认 stub，配 `HIKVISION_URL` 走真实海康 ISAPI。**与模型无关**（执行器适配器）。

## H. 门卫数据助手：对话 + 结构化筛选"双模"，且查询工具是安全只读的

- **为什么**：加分项要求保安能用自然语言查数据；但用户也要求"不只是对话，机械筛选也要全面"；同时不能让查询变成注入风险。
- **优势**：
  - **对话模式**（`/ask` 💬）：自然语言多轮追问（"这个月放行多少？"→"那上周呢？"）。
  - **筛选模式**（`/ask` 🔎）：今天/本周/本月 + 单位/车牌/状态 → 确定性出数量/已放行数/时段直方图/结果表，**不走 LLM**。
  - **电话查询**：门卫白名单号码拨入→语音数据助手（复用同一套工具）。
- **根因 + 安全**：LLM 只能调用**参数化的只读工具**（`count_visits`/`list_visits`/`busiest_hours`/`frequent_visitors`），**不是 text-to-SQL**——模型选工具+填参数，查询由我执行。
- **【与模型关系】这是"既相关又无关"的范例**：
  - 对话查询依赖 LLM 的工具调用能力（**相关**）；
  - 结构化筛选是确定性 SQL（**无关**）——即使没有模型/模型不可靠，保安仍能用筛选拿到全部数据；
  - **安全性与模型无关**：即便模型"乱来"，它能执行的也只有白名单只读工具，碰不到任意 SQL。**我把"能执行什么"的边界从模型手里收回。**

## I. 鉴权与角色区分：访客不能查数据（口令 + 手机白名单）

- **为什么**：门卫可查数据，访客绝不能；网页和电话两条路都要能区分。
- **优势**：
  - 网页 `GUARD_ACCESS_KEY`：中间件守 `/dashboard /ask /admin` + 数据 API；访客页 `/voice /qr` 与带 token 的 `/confirm` 仍公开。
  - 电话 `GUARD_PHONES`：来电号码在白名单→语音数据助手；否则→访客登记。
- **根因**：网页是一段鉴权中间件；电话是按 caller-ID 路由到不同的 Agent persona。默认两者留空=零行为变化，设了才生效。
- **【与模型关系】无关**。鉴权与路由是确定性规则，不依赖模型。

## J. 数据存储与持久化

- **J1 SQLite 本地 / Neon Postgres 云，一行 `DATABASE_URL` 切换**：demo 用文件、生产用云库，代码零改。**与模型无关**。
- **J2 持久化修复（相对路径锚定项目根）**
  - 为什么：用户每次测试数据像被清空、回访识别不到老访客。根因是 `sqlite:///./data/visits.db` 相对**当前启动目录**，Windows 下从不同目录启动 python→命中不同的空库。
  - 优势：agent + web + 每次运行**共用同一个 DB 文件**，回访识别才连续；启动日志打印绝对路径。
  - 根因：把相对 sqlite 路径锚定到项目根目录（`_resolve_sqlite_url`）。
  - 【与模型关系】**无关**（纯工程健壮性）。

## K. 部署：本地 → 全云端（Dockerfile + Fly/Railway + LiveKit Cloud + Neon）

- **为什么**：要能"彻底甩掉家用电脑"。
- **优势**：worker 外连 LiveKit Cloud、数据进 Neon、容器只暴露 HTTP 端口，可全托管。**与模型无关**（部署形态）。

## 总结：模型相关性的三层结论（答辩一句话版）

1. **关键属性我刻意做成"与模型无关"**：并发、安全/鉴权、封闭集合的数据准确性、查询安全、持久化、三接入统一、放行闭环——靠框架与确定性算法保证，换任何 LLM 都成立。这是项目的**稳定底座**。

2. **少数体验性优势"与模型相关"**：realtime 低延迟、自然对话、复述确认、对话式查询——这些用 LLM 才好。我把它们用在模型擅长的地方，但**都配了确定性兜底或回退**（realtime↔pipeline、复述靠人确认、对话查询旁边有结构化筛选、查询限定在只读工具内）。

3. **"模型可换"是把第 2 类从风险变成红利的元设计**：模型越强，相关优势越好；某模型不合规/不可用，一行切换、主流程不动。**所以"跟模型相关"在这里不是脆弱点，而是可升级、可回退、不锁定的可控变量。**

> 一句话：**我用工程和确定性算法守住"准确、安全、可靠、并发"，把模型用在"自然与低延迟"上，并让模型本身可随时替换——优势该靠架构的归架构，该靠模型的归模型，且两者解耦。**
